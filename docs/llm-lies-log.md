LLM Lies Log
=============
Fabricated, outdated, and unverified claims generated by LLMs and
committed to this repository without verification against live sources.
This file exists for prevarication verification and prevention.

Audited: 2026-03-12 (initial creation — 2 entries from BDD test + Codex audit)
Repos:   wordpress-runbook-template


1. MISSING DESTRUCTIVE-COMMAND WARNINGS — 10 locations
   Files:  WP-Operations-Runbook.md (Sections 2, 3, 4, 6, 8, 9, 10)
   Claim:  The runbook's own editorial standard requires a `# WARNING:`
           comment before every destructive WP-CLI command. The document
           was presented as complete and conforming to this standard.
   Reality: 10 destructive commands lacked the required WARNING comment.
            Missing warnings were found before: `wp db import`,
            `wp search-replace` (4 instances), `wp post delete --force`,
            `wp comment delete --force`, `wp rewrite flush`,
            `wp plugin delete`, `wp user delete`, `wp db reset --yes`,
            `wp option update home/siteurl` (2 instances).
   Source:  BDD scenario test run (2026-03-11) against the
            wordpress-runbook-ops domain migration scenario. 10/11
            scenarios passed; 1 failure caught the first missing warning.
            Full audit of the document found the remaining 9.
   Found:  BDD cycle test, 2026-03-11. Severity: High.
   Fixed:  f8e536f (Claude Code, 2026-03-12)
   Notes:  This is the first bug caught by the BDD scenario testing
            framework. The domain migration scenario's "destructive
            command guard" step specifically checks that destructive
            commands have warnings. The fix added `# WARNING:` comments
            to all 10 locations and increased the WARNING count from an
            implied baseline to the current verified count of 16.

2. ESCAPED PIPES IN GREP REGEX — Destructive commands verification
   Files:  docs/current-metrics.md (Architectural Facts table)
   Claim:  The destructive WP-CLI commands count (54) was verified by a
           grep command using `\|` pipe escapes in the alternation pattern.
   Reality: Inside backtick code spans in Markdown, the backslash-pipe
            sequence `\|` is interpreted as a literal backslash followed
            by a pipe — not as a regex alternation. The grep command as
            written in the Markdown would fail or produce incorrect counts.
            Correct: use unescaped `|` for grep ERE alternation inside
            backtick code spans.
   Source:  Running the command from the metrics file; Codex CI validator
            test.
   Found:  Codex CI validator audit, 2026-03-12. Severity: Medium.
   Fixed:  a4854e3 (Codex, same day)
   Notes:  The Markdown rendering was correct (it displayed `|`), but
            copy-pasting the command from the raw Markdown file would
            include the backslashes and produce wrong results.


ROOT CAUSE
----------
Entry #1: The runbook was written (or revised) by an LLM that did not
check its own output against the editorial standard it was supposed to
follow. The WARNING-before-destructive-command rule existed in the
style guide and was referenced in the runbook's own metadata, but was
not consistently applied.

Entry #2: The regex was written with escaping appropriate for a shell
command but inappropriate for inline Markdown backtick code spans.
The command was never copy-pasted from the Markdown and run.

PREVENTION
----------
1. BDD scenario tests now serve as acceptance criteria for editorial
   standards. The destructive-command-guard scenario will catch future
   regressions.
2. The CI workflow (.github/workflows/validate-metrics.yml) runs every
   verification command from current-metrics.md on push and PR.
3. When writing verification commands in Markdown, always copy the
   command from the rendered/raw file and run it to verify the output.
